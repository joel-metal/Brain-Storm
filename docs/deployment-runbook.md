# Deployment Runbook

Step-by-step guide for deploying Brain-Storm to production. Follow sections in order. Do not skip the pre-deployment checklist.

**Audience:** Engineers with production access  
**Estimated time:** 30–45 min (first deploy), 15–20 min (subsequent releases)

## 1. Pre-Deployment Checklist

Complete every item before touching production. Check off as you go.

### Code & CI

- [ ] All CI checks green on the release commit (`ci.yml` — backend tests, frontend build, contract checks)
- [ ] Release PR reviewed and merged to `main`
- [ ] GitHub Release created and tagged (e.g. `v1.2.0`) — this triggers `deploy-production.yml`
- [ ] `CHANGELOG.md` updated (auto-generated by release-please)
- [ ] No open `P0`/`P1` bugs against this release

### Database

- [ ] Database backup taken and verified restorable:
  ```bash
  pg_dump -h $DATABASE_HOST -U $DATABASE_USER brain-storm \
    | gzip > backup-$(date +%Y%m%d-%H%M%S).sql.gz
  ```
- [ ] Pending migrations reviewed (`apps/backend/src/migrations/`)
- [ ] Migrations tested on staging with a production-size dataset
- [ ] Confirm `synchronize: false` in `data-source.ts` (never auto-sync in production)

### Secrets & Environment

- [ ] All required env vars present in the production secret store (see table below)
- [ ] `JWT_SECRET` is at least 64 random bytes
- [ ] `STELLAR_SECRET_KEY` is the correct mainnet issuer keypair
- [ ] Contract IDs in env match `scripts/deployed-contracts.json` for `mainnet`

**Required production env vars:**

| Variable | Notes |
|---|---|
| `DATABASE_HOST` / `PORT` / `USER` / `PASSWORD` / `NAME` | PostgreSQL connection |
| `REDIS_URL` | `redis://host:6379` |
| `JWT_SECRET` | Min 64 chars, random |
| `STELLAR_NETWORK` | Must be `mainnet` |
| `STELLAR_SECRET_KEY` | Mainnet issuer keypair |
| `SOROBAN_RPC_URL` | `https://soroban.stellar.org` |
| `ANALYTICS_CONTRACT_ID` | From `deployed-contracts.json` |
| `TOKEN_CONTRACT_ID` | From `deployed-contracts.json` |
| `CERTIFICATE_CONTRACT_ID` | From `deployed-contracts.json` |
| `GOVERNANCE_CONTRACT_ID` | From `deployed-contracts.json` |
| `NEXT_PUBLIC_API_URL` | Public backend URL |
| `SENTRY_DSN` / `NEXT_PUBLIC_SENTRY_DSN` | Error tracking |
| `EMAIL_HOST` / `EMAIL_USER` / `EMAIL_PASS` | Transactional email |
| `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` | OAuth |

### Capacity & Dependencies

- [ ] Stellar mainnet status green: https://status.stellar.org
- [ ] Confirm PostgreSQL has sufficient disk space (>20% free)
- [ ] Redis memory usage below 70%
- [ ] Notify the team in `#deployments` Slack channel: `"Deploying vX.Y.Z to production at HH:MM UTC"`

## 2. Deploy: Smart Contracts

Only required when contract source has changed. Skip this section for backend/frontend-only releases.

### 2a. Build WASM

```bash
# From repo root
rustup target add wasm32-unknown-unknown   # if not already added
./scripts/build.sh
```

Verify output — each contract should produce a `.wasm` file under `target/wasm32-unknown-unknown/release/`:

```
brain_storm_analytics.wasm
brain_storm_token.wasm
brain_storm_certificate.wasm
brain_storm_governance.wasm
brain_storm_shared.wasm
```

### 2b. Run Contract Tests

```bash
cargo test
```

All tests must pass before deploying to mainnet.

### 2c. Deploy Each Contract

```bash
export STELLAR_SECRET_KEY=<mainnet-issuer-secret>

./scripts/deploy.sh mainnet analytics
./scripts/deploy.sh mainnet token
./scripts/deploy.sh mainnet certificate
./scripts/deploy.sh mainnet governance
./scripts/deploy.sh mainnet shared
```

Each command prints the new `Contract ID` and updates `scripts/deployed-contracts.json` automatically.

### 2d. Update Environment Variables

Copy the new contract IDs from `scripts/deployed-contracts.json` into the production secret store:

```bash
cat scripts/deployed-contracts.json | jq '.mainnet'
```

Update `ANALYTICS_CONTRACT_ID`, `TOKEN_CONTRACT_ID`, `CERTIFICATE_CONTRACT_ID`, `GOVERNANCE_CONTRACT_ID`, and `SHARED_CONTRACT_ID` in your secrets manager before proceeding.

### 2e. Initialize New Contracts (first deploy only)

If deploying a contract for the first time, call `initialize` with the admin address. See [smart-contract-interaction-guide.md](./smart-contract-interaction-guide.md) for the TypeScript helper. Example using Stellar CLI:

```bash
stellar contract invoke \
  --id <CONTRACT_ID> \
  --source $STELLAR_SECRET_KEY \
  --network mainnet \
  -- initialize \
  --admin <ADMIN_STELLAR_ADDRESS>
```

> **Warning:** `initialize` can only be called once. Double-calling panics with `"Already initialized"`.

## 3. Deploy: Backend

### Automated Path (recommended)

Push a GitHub Release tag — `deploy-production.yml` handles the rest:

1. Builds and pushes the Docker image to GHCR with the release tag
2. Requests production environment approval (GitHub Environments gate)
3. Runs database migrations inside the new image
4. Deploys the new image
5. Runs health checks (30 attempts × 10s = 5 min window)
6. Auto-rolls back on failure

Monitor the workflow at: `https://github.com/BrainTease/Brain-Storm/actions`

### Manual Path

Use this if the automated workflow is unavailable.

**Step 1 — Pull the release image**

```bash
docker pull ghcr.io/braintease/brain-storm/backend:v1.2.0
```

**Step 2 — Run database migrations**

```bash
docker run --rm \
  --env-file .env.production \
  ghcr.io/braintease/brain-storm/backend:v1.2.0 \
  node dist/main migration:run
```

Confirm output ends with `migrations successfully ran` and no errors.

**Step 3 — Deploy**

Using `docker-compose.prod.yml`:

```bash
IMAGE_TAG=v1.2.0 docker compose -f docker-compose.prod.yml up -d --no-deps backend
```

Or for ECS/Kubernetes, update the task definition / deployment image tag to the new version.

**Step 4 — Verify the container started**

```bash
docker compose -f docker-compose.prod.yml ps backend
docker compose -f docker-compose.prod.yml logs --tail=50 backend
```

Look for: `NestApplication successfully started` and no `ERROR` lines.

## 4. Deploy: Frontend

The frontend is a Next.js static/SSR app. It is built and deployed independently of the backend.

**Step 1 — Build**

```bash
NEXT_PUBLIC_API_URL=https://api.brain-storm.example.com \
NEXT_PUBLIC_STELLAR_NETWORK=mainnet \
NEXT_PUBLIC_GIT_COMMIT_SHA=$(git rev-parse HEAD) \
npm run build --workspace=apps/frontend
```

**Step 2 — Deploy**

Using `docker-compose.prod.yml`:

```bash
IMAGE_TAG=v1.2.0 docker compose -f docker-compose.prod.yml up -d --no-deps frontend
```

Or push the built `.next/` output to your CDN / Vercel / static host.

**Step 3 — Verify**

```bash
curl -sf https://brain-storm.example.com | grep -q "Brain-Storm" && echo "✅ Frontend up"
```

## 5. Health Check Verification

Run these checks immediately after every deployment, in order.

### 5a. Backend Health Endpoint

```bash
API_URL=https://api.brain-storm.example.com \
MAX_RETRIES=30 \
RETRY_INTERVAL=10 \
./scripts/health-check.sh
```

Expected response (`HTTP 200`):

```json
{
  "status": "ok",
  "info": {
    "database": { "status": "up" },
    "redis":    { "status": "up" },
    "stellar":  { "status": "up" }
  }
}
```

A `503` means one or more dependencies are unhealthy — check the `details` field.

### 5b. Smoke Test Key Endpoints

```bash
BASE=https://api.brain-storm.example.com

# Auth
curl -sf -o /dev/null -w "%{http_code}" $BASE/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"smoke@test.invalid","password":"x"}' | grep -q 401 && echo "✅ Auth endpoint"

# Courses list
curl -sf -o /dev/null -w "%{http_code}" $BASE/v1/courses | grep -q 200 && echo "✅ Courses endpoint"

# Swagger docs
curl -sf -o /dev/null -w "%{http_code}" $BASE/api/docs | grep -q 200 && echo "✅ Swagger docs"

# Metrics
curl -sf -o /dev/null -w "%{http_code}" $BASE/metrics | grep -q 200 && echo "✅ Metrics endpoint"
```

### 5c. Stellar Connectivity

```bash
curl -sf https://api.brain-storm.example.com/v1/stellar/network-status | jq '.horizon.status, .soroban.status'
```

Both should return `"healthy"`.

### 5d. Frontend

```bash
curl -sf https://brain-storm.example.com | grep -q "Brain-Storm" && echo "✅ Frontend serving"
```

### 5e. Database Migration Confirmation

```bash
psql -h $DATABASE_HOST -U $DATABASE_USER -d brain-storm \
  -c "SELECT version, name, \"executedAt\" FROM migrations ORDER BY \"executedAt\" DESC LIMIT 5;"
```

Confirm the latest migration timestamp matches what was just deployed.

## 6. Rollback Procedures

### Automated Rollback

`deploy-production.yml` automatically rolls back to the previous GitHub Release tag if the health check fails. A comment is posted on the failed release with the rollback target version.

### Manual Backend Rollback

**Step 1 — Identify the previous stable image tag**

```bash
# List recent GHCR images
gh api /orgs/BrainTease/packages/container/brain-storm%2Fbackend/versions \
  --jq '.[0:5] | .[] | .metadata.container.tags[]'
```

**Step 2 — Redeploy the previous image**

```bash
IMAGE_TAG=v1.1.3 docker compose -f docker-compose.prod.yml up -d --no-deps backend
```

**Step 3 — Revert migrations if the new migration is destructive**

```bash
# Inside the previous image container
npm run typeorm:revert --workspace=apps/backend
```

> Only revert if the migration added columns/tables. Never revert a migration that dropped data — restore from backup instead.

**Step 4 — Verify health**

Re-run [Section 5](#5-health-check-verification).

### Manual Frontend Rollback

```bash
IMAGE_TAG=v1.1.3 docker compose -f docker-compose.prod.yml up -d --no-deps frontend
```

### Manual Contract Rollback

Soroban contracts are immutable once deployed. "Rollback" means repointing the backend env vars to the previous contract ID:

1. Find the previous contract ID in git history: `git log --oneline scripts/deployed-contracts.json`
2. Update `ANALYTICS_CONTRACT_ID` (and others) in the production secret store to the previous values
3. Redeploy the backend with the updated env vars

If the contract has an `upgrade` function (Shared contract does), you can also redeploy the previous WASM:

```bash
stellar contract install --wasm target/wasm32-unknown-unknown/release/brain_storm_shared.wasm \
  --source $STELLAR_SECRET_KEY --network mainnet
# Then invoke upgrade() with the old wasm hash
```

### Rollback Decision Matrix

| Symptom | Action |
|---|---|
| Health check fails, no DB migration | Redeploy previous backend image |
| Health check fails after migration | Redeploy previous image + revert migration |
| Frontend 500s | Redeploy previous frontend image |
| Contract calls failing | Repoint env vars to previous contract IDs + redeploy backend |
| DB corruption | Restore from pre-deployment backup |

## 7. Post-Deployment Monitoring

### Immediate (first 30 minutes)

Watch these in real time after every deployment:

**Error rate** — should stay below 1% of requests:
```bash
# Prometheus query
rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m])
```

**Application logs:**
```bash
docker compose -f docker-compose.prod.yml logs -f backend | grep -E "ERROR|WARN"
```

**Sentry** — check for new error groups at https://sentry.io. Any new `P0` errors warrant immediate investigation.

### Dashboards

Start the monitoring stack if not already running:

```bash
docker compose -f docker-compose.monitoring.yml up -d
```

| Dashboard | URL | What to watch |
|---|---|---|
| Grafana | http://localhost:3002 (admin/admin) | HTTP rates, credential issuance, BST minting, Stellar RPC latency |
| Prometheus | http://localhost:9090 | Raw metrics, alert rules |

Key metrics to verify are healthy post-deploy:

| Metric | Healthy threshold |
|---|---|
| `http_requests_total` (5xx rate) | < 1% |
| `stellar_rpc_latency_seconds` p95 | < 2s |
| `nodejs_eventloop_lag_seconds` | < 100ms |
| `process_resident_memory_bytes` | Stable (not growing) |
| `credential_issued_total` | Incrementing normally |

### Alerting Thresholds

Configure these Prometheus alert rules in `infra/monitoring/prometheus.yml`:

```yaml
groups:
  - name: brainstorm
    rules:
      - alert: HighErrorRate
        expr: rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m]) > 0.01
        for: 2m
        annotations:
          summary: "Error rate above 1%"

      - alert: StellarRpcSlow
        expr: histogram_quantile(0.95, stellar_rpc_latency_seconds_bucket) > 2
        for: 5m
        annotations:
          summary: "Stellar RPC p95 latency above 2s"

      - alert: BackendDown
        expr: up{job="brain-storm-backend"} == 0
        for: 1m
        annotations:
          summary: "Backend is unreachable"
```

### 24-Hour Follow-Up

- [ ] Confirm no new Sentry error groups opened overnight
- [ ] Check PostgreSQL slow query log for regressions
- [ ] Verify `credential_issued_total` and `bst_minted_total` counters are incrementing as expected
- [ ] Post deployment summary in `#deployments`: version, deploy time, any issues encountered

### Escalation

| Severity | Condition | Action |
|---|---|---|
| P0 | Backend down / health check failing | Page on-call engineer immediately, begin rollback |
| P1 | Error rate > 5% sustained > 5 min | Notify team, assess rollback |
| P2 | Stellar RPC degraded, non-fatal | Monitor, open incident ticket |
| P3 | Elevated latency, no errors | Monitor for 30 min, escalate if worsening |
